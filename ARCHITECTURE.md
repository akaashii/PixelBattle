# Pixel Battle — Архитектура

## Стек технологий

| Компонент | Технология | Почему |
|-----------|-----------|--------|
| Telegram Bot | **aiogram 3.x** | Асинхронный, нативная поддержка WebApp-кнопок |
| Backend API + WebSocket | **FastAPI + uvicorn** | Один процесс = HTTP + WS; Pydantic-валидация «из коробки» |
| Хранилище холста | **Redis** (битовый массив) | O(1) чтение/запись пикселя; `GETRANGE` для bulk-загрузки |
| Хранилище игр/юзеров | **PostgreSQL** (через asyncpg) | Надёжно для мета-данных: игры, команды, статистика |
| Кулдаун | **Redis** (`SET EX 60`) | Атомарная проверка + установка TTL за одну операцию |
| Синхронизация клиентов | **WebSocket** (FastAPI) | Реалтайм-пуш дельт; Long Polling здесь неприемлем |
| Frontend | **Vanilla JS + Canvas API** | Минимальный размер; Telegram WebApp SDK подключается через CDN |

## Схема взаимодействия

```
Telegram ──► aiogram Bot ──► PostgreSQL (игры, команды)
                │
                ▼
         Inline-кнопка WebApp URL
                │
                ▼
   Браузер (Telegram WebApp)
       │               ▲
       │  HTTP GET      │  WebSocket (дельты)
       ▼               │
    FastAPI Backend ◄──►  Redis
      (API + WS)         ├─ canvas:{game_id}  (бинарный массив)
                         ├─ cooldown:{game_id}:{user_id}
                         └─ game:{game_id}:meta
```

## Хранение холста в Redis

Поле 100×100 = 10 000 пикселей.  
Каждый пиксель = **1 байт** (0 = пусто, 1 = красные, 2 = синие).  
Итого: **10 KB** на игру — мгновенное чтение одним `GET`.

Для 500×500 = 250 000 байт = **250 KB** — всё ещё приемлемо.

### Запись пикселя (атомарность)

```
SETRANGE canvas:{game_id} <offset> <team_byte>
```

`offset = y * WIDTH + x`. Redis гарантирует атомарность `SETRANGE` →  
гонки при одновременной записи в **одну координату** разрешаются  
по принципу «последний пишет» (last-write-wins), что для Pixel Battle  
является корректным поведением.

## Кулдаун (защита от читерства)

```python
key = f"cd:{game_id}:{user_id}"
# SET key 1 NX EX 60  →  вернёт True только если ключа не было
ok = await redis.set(key, 1, nx=True, ex=60)
if not ok:
    ttl = await redis.ttl(key)
    raise CooldownError(ttl)
```

`NX` + `EX` — единая атомарная команда, гонки невозможны.

## Синхронизация (WebSocket)

1. Клиент открывает WS → сервер добавляет в `set` подписчиков игры.
2. При записи пикселя сервер делает **broadcast** дельты всем подписчикам:
   ```json
   {"type":"pixel","x":42,"y":17,"team":1,"user_id":123}
   ```
3. Клиент применяет дельту к локальному Canvas — **перезагрузка не нужна**.

## Жизненный цикл игры

1. `/start_battle` → создание записи в PostgreSQL, статус `RECRUITING`.
2. 2 минуты сбор → деление на команды, статус `ACTIVE`, запуск 24h таймера.
3. Каждые 2 часа → бот отправляет статистику в чат.
4. Через 24 часа → статус `FINISHED`, подсчёт пикселей, объявление победителя.
